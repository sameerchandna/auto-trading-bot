Auto Trading Bot — User Manual

EURUSD-first multi-asset bot with multi-timeframe confluence analysis, walk-forward validation, and self-learning parameter research. Paper trading on an OANDA practice account; ~£10k starting capital.

The pipeline in one breath

Data ingestion → multi-TF analysis (15m/1h/4h/1d/1w) → confluence scoring → signal generation → risk management → execution → learning. Everything's driven by config/optimized_params.json (single source of truth, synced to PineScript via /update-strategy).

Results of python main.py backtest runs — one row per run, with win rate, P&L, Sharpe, drawdown, and long/short breakdown. Fold-run children are filtered out so this view stays clean.

Walk-forward fold runs (backtest-folds). List view shows % profitable folds, combined OOS P&L, combined profit factor, combined DD, mean Sharpe. Click a row for per-fold detail, a combined OOS equity curve, and (for optimized runs) a param-drift table.
Research
Automated parameter sweeps run by the research agent. Each row is a mutation tested across scenarios with median PF/WR/DD, OOS check, and a verdict. Budget/baseline state at the top.
Comparisons
A/B comparison groups — baseline runs paired with variants to visualize rule impact.
Learning
Parameter history + current production values. Surfaces how params have drifted over time as the learner promotes improvements.
Data
Data-health view: candle coverage per pair/timeframe, last-updated timestamps, gaps.

What happens when a backtest runs

  1. Load candles from SQLite cache for the requested date range. If missing, fetch from Yahoo and cache.
  2. Iterate 1H candles (starting after a 50-candle warmup). At each bar, build a multi-TF context by binary-searching a 200-candle window ending strictly before the current bar (no lookahead).
  3. Score confluence (HTF bias, BOS, wave, liquidity sweep, S/R, wave ending) using the production params.
  4. Risk-manage signals that clear the threshold: size at 2% risk, enforce max 3 concurrent positions, daily/weekly loss caps.
  5. Open/close positions on SL/TP hits. Record every trade to positions (tagged with a bt_<id>) and journal entries.
  6. Compute 19 metrics (Sharpe/Sortino on daily-resampled returns, PF, expectancy, DD, etc.) and persist to backtest_runs.
  7. Render results in the Backtests tab.

What happens when a fold run runs

Phase 1 (validation-only, no --optimize):

  1. Slice the date range into rolling quarterly folds (14 for 2023-01-01 → today). Final in-progress quarter is marked partial and excluded from summary stats.
  2. Fan out folds across ProcessPoolExecutor workers — each worker constructs its own BacktestEngine for its fold's OOS window with baseline params.
  3. Per-fold metrics + equity curves come back; aggregator concatenates segments into a combined OOS curve (each fold rebased to the prior fold's ending equity).
  4. Summary: mean/median/std/min/max across folds for Sharpe, PF, WR, DD, P&L; % profitable folds; combined metrics over the concatenated curve.
  5. Persist to backtest_folds_runs (parent) + tagged child rows in backtest_runs with fold_parent_id set.

Phase 2 (with --optimize --trials N):

  1. For each fold, the worker first runs Optuna on the fold's IS window (all prior data within range), fitting weights, threshold, SL multiplier, TP R:R, swing lookback. Folds with IS < 180 days fall back to baseline params.
  2. The best params are then frozen and evaluated on that fold's OOS — a genuine out-of-sample test of the optimization process, not of any single param set.
  3. Report adds IS PF / OOS PF / ΔPF columns, an IS→OOS degradation panel (mean Sharpe/PF/WR drop), and a param-drift block showing how the optimizer's choices move across folds.

Why this matters: optimizing once on all data tells you "these params look good on data they were fit to" — every fold is in-sample and OOS numbers are fiction. Per-fold walk-forward is the honest version and also mirrors how deployment actually works (refit as new data arrives).

Key guardrails

  • Max 2% risk per trade, max 3 concurrent positions.
  • Daily loss limit 4%, weekly 8%.
  • Paper trading only until backtests prove positive expectancy over 200+ trades.
  • BT ↔ PineScript params must stay in sync — enforce with python -m tradingview.generate_pine --check.

Where things live in the DB

  • candles — raw OHLCV cache
  • positions — every open/closed trade (live + backtest, distinguished by tags)
  • signals — emitted signals (not all become positions)
  • backtest_runs — single-range backtest results (with fold_parent_id linking fold children)
  • backtest_folds_runs — parent row per fold run with summary/per-fold/drift/combined curve JSON
  • trade_journal — post-trade analysis snapshots
  • parameters — historical param promotions by the learner
