Auto Trading Bot — User Manual

EURUSD-first multi-asset bot with multi-timeframe confluence analysis, walk-forward validation, and self-learning parameter research. Paper trading on an OANDA practice account; ~£10k starting capital.

Pipeline: Data ingestion → multi-TF analysis (15m/1h/4h/1d/1w) → confluence scoring → signal generation → risk management → execution → learning. Single source of truth: config/optimized_params.json (synced to PineScript via /update-strategy).

Guardrails & rules

  • Max 2% risk per trade
  • Max 3 concurrent positions
  • Daily loss limit 4%, weekly 8%
  • Paper trading only until 200+ trades prove positive expectancy
  • BT ↔ PineScript params kept in sync (generate_pine.py --check)
  • No lookahead: candle windows are strictly bar-closed

What a backtest does

  1. Load candles from SQLite cache (fetch from Yahoo if missing).
  2. Iterate 1H candles after a 50-bar warmup. Build multi-TF context via binary-searched 200-bar windows ending strictly before the current bar.
  3. Score confluence (HTF bias, BOS, wave, sweep, S/R, ending) with production params.
  4. Risk-manage: 2% sizing, max 3 open, daily/weekly caps.
  5. Open/close on SL/TP. Tag positions as bt_<id>, journal each trade.
  6. Compute 19 metrics (Sharpe/Sortino daily-resampled, PF, expectancy, DD…).
  7. Persist to backtest_runs; show in Backtests tab.

What a fold run does — Phase 1 (validate)

  1. Slice date range into rolling quarterly folds (14 for 2023-01-01 → today). Current quarter is partial and excluded from summary.
  2. Fan folds across ProcessPoolExecutor workers — each builds its own engine for its OOS window with baseline params.
  3. Aggregator concatenates per-fold curves (rebased) into a combined OOS curve.
  4. Summary: mean/median/std/min/max across folds; % profitable; combined metrics.
  5. Persist parent to backtest_folds_runs; children to backtest_runs with fold_parent_id.

What a fold run does — Phase 2 (--optimize)

  1. Per fold, run Optuna on the IS window (all prior data). Folds with IS < 180 days fall back to baseline.
  2. Freeze the best params, then evaluate on OOS — honest out-of-sample test of the optimization process.
  3. Report adds IS PF / OOS PF / ΔPF columns, IS→OOS degradation block, and param drift across folds.

Why per-fold: optimizing once on all data makes every fold in-sample and OOS numbers fiction. Per-fold walk-forward also mirrors how deployment works — refit as new data arrives.
